問1　Web APIのCORSとSame-Originポリシー

設問1(1)
解答例
a＝Same-Origin（同一オリジンポリシー）
ブラウザのセキュリティ基本機能。オリジン（スキーム＋ホスト＋ポート）が異なるリソースへのアクセスをJavaScriptから禁止する。CORSはこの制限を安全に緩和するためのHTTPヘッダベースの仕組み。
設問1(2)
解答例
b＝イ・キ（CORSヘッダの設定値） c,d＝ク（許可するOrigin）
CORSのAccess-Control-Allow-Originヘッダ。特定のオリジンのみ許可する場合は具体的なオリジンを指定（'*'は認証情報を含むリクエストには使えない）。許可オリジンをホワイトリスト管理することが重要。
設問2
解答例
WebサイトBへのログイン（CSRF等の悪用可能性）
CORSの過剰許可によるリスク。Access-Control-Allow-Originsに悪意あるサイトのオリジンを許可した場合、そのサイトのJavaScriptから認証済みAPIを呼び出せる。XMLHttpRequestのwithCredentials=trueと組み合わさると深刻。

問2　クラウドサービスの認証強化とFIDO

設問1(1)
解答例
ホテルWi-Fiと同じSSIDと事前共有鍵（偽AP設置による中間者攻撃）
Evil Twin（偽AP）攻撃。正規のホテルWi-FiのSSIDと事前共有鍵を模倣した偽APを設置し、被害者のデバイスを接続させて通信を傍受・改ざんする。VPNの使用と証明書検証による防止が有効。
設問1(3)
解答例
HTTP で接続が開始されたから（HTTPSへの強制リダイレクトなし）
SSLストリッピング攻撃。HTTPでの接続開始時にHTTPSへのリダイレクトをインターセプトし、HTTPのまま通信を維持する攻撃。HSTSプリロードにより初回からHTTPS強制することで防止できる。
設問2(1)
解答例
OTPの入力を要求し、OTPを認証サーバへ中継する処理
リアルタイムフィッシング（OTPリレー攻撃）。攻撃者の偽サイトがユーザ入力のOTPを即座に正規サーバに転送する。FIDO2/WebAuthnはオリジンを認証に含めるためこの攻撃に耐性がある。
設問2(2)
解答例
f＝認証サーバがオリジンbとオリジンsの一致を確認しているから（FIDO2のオリジン検証）
WebAuthn（FIDO2）のフィッシング耐性の仕組み。認証時にブラウザがRPID（origin）を含めた署名を生成し、サーバが登録時のオリジンと一致するか検証する。偽サイトでは異なるオリジンとなり認証が失敗する。

問3　IoT機器のセキュリティ設計と認証連携

設問1
解答例
エ（ゲーム機のセキュリティリスク）
IoT機器の物理的アクセスリスク。ゲーム機は物理的に手元にあるためハードウェアデバッグインターフェース（JTAG・UART）を悪用した解析・改ざんが可能。セキュアブートと耐タンパ性が対策。
設問2(2)
解答例
ゲームサーバに認証サーバと同じ共通鍵を保存する
共通鍵（対称鍵）を使ったトークン検証の仕組み。認証サーバが発行したJWT（HMAC署名）をゲームサーバが同じ鍵で検証する。鍵を共有するため鍵漏えいの影響範囲が広い。非対称鍵（JWT RS256）の方がセキュリティ上望ましい。
設問2(5)
解答例
耐タンパ性（Tamper Resistance）
ハードウェアセキュリティの重要概念。SSDを取り出して解析されても内部のデータ（秘密鍵・DRMキー等）を保護するための物理的・論理的な防護技術。TPM・HSM・Secure Enclaveが実装例。
