問1　コンサルティング業務で利用するSaas

H社では、コンサルティング業務で利用する Saas（以下、Sサービスという）を提供している。コンサルティング業務を行う企業 50社程度がSサービスを利用しており、総利用者数は1,000名程度である。
[Sサービスの概要]
Sサービスでは、利用企業ごとに複数のプロジェクトを登録できる。Sサービスの利用者には2種類のロールがある。自社の全てのプロジェクトの情報にアクセスできる管理者と、自身に割り当てられているプロジェクトの情報だけにアクセスできる一般利用者である。
Sサービスには、プロジェクトを管理する機能（以下、プロジェクト管理機能という）のほか、利用者がプロジェクト内で特定の話題ごとに“スレッド”と呼ばれるページを作成する機能（以下、スレッド作成機能という）、スレッドに文章を投稿したりファイルをアップロードしたりする機能（以下、スレッド投稿機能という）などが設けられている。

問1　WebアプリケーションのインシデントとXSS・CSRF対策


設問1
解答例
a＝できない　b＝できない
図1のCSPは `Content-Security-Policy: default-src 'self'`。`default-src 'self'` はリソースの読み込みを同一オリジンのみに制限し、`'unsafe-inline'` も `'nonce-'` も含まれないため、インラインスクリプトも外部ドメインリソースも両方実行・読み込みできない。これが後述の攻撃成功の鍵となる—最初の失敗した攻撃は inline script `<script>悪意コード</script>` 形式だったためCSPに遮断された。


設問2(1)
解答例
ロール管理
図3のスクリプトは①GET /management/role でcsrf_tokenを取得 → ②POST /management/roleset でis_admin=1&user_id=U803318019 を送信することで、任意の利用者を管理者に昇格させる。これは表1の「ロール管理」機能を悪用した CSRF/権限昇格攻撃。スクリプトがSサービス内からfetchするためSame-Originポリシーは問題なく、csrf_tokenも同一オリジンのGETで取得できてしまう。


設問2(2)
解答例
タスク名
格納型XSS（Stored XSS）。タスク名の入力値が出力時にエスケープされずHTMLに埋め込まれるため、`<script src=...>` タグが有効なHTML要素として解釈される。出力される画面はプロジェクト進捗管理（期限切れタスク一覧）。攻撃者が登録したタスク名がそのまま `<li>` 内に展開されている点が図2から確認できる。


設問2(3)
解答例
個人タスク　＜script src="/files/F1234567890.xlsx"＞＜/script＞
図2の該当行: `<li>個人タスク<script src="/files/F1234567890.xlsx"></script>（締切日過ぎ5日）</li>`。スクリプトのsrcに `/files/` パス（Sサービスと同一オリジン）を使うことで `default-src 'self'` のCSPを通過させるのが工夫のポイント。ファイル識別子を変更しても拡張子.xlsxは保持されるため、JSとして読み込まれる。


設問2(4)
解答例
e＝管理者　f＝タスクの締切日を過ぎる　g＝利用者のロールを管理者に設定する
攻撃フローの整理: e（誰が閲覧）＝プロジェクトに参加している管理者。管理者は全スレッド・全プロジェクトにアクセスでき、進捗管理画面を定期的に確認する立場。f（表示条件）＝未完了かつ締切日超過でリストに表示される仕組み。g（実行された処理）＝U803318019のis_admin=1への変更→元従業員Zが管理者になる。その後、管理者権限で全プロジェクトのファイルダウンロードが可能に。


設問2(5)
解答例
スクリプトをSサービス内の /files/ パスにアップロードし、src属性でそのURLを指定すること（同一オリジンにすることでCSPを通過させる）
最初の失敗した試みはおそらく `<script>インラインコード</script>` または外部ドメインのsrc指定。CSP `default-src 'self'` は(a)インラインスクリプト実行と(b)外部オリジンからの読み込みを両方禁止する。工夫は「スクリプトをSサービス自体にファイルとしてアップロードし、/files/[id].xlsx という同一オリジンURLをsrcに指定する」こと。MIMEタイプのチェック不備により、拡張子.xlsxでもブラウザがJSとして実行してしまった。


設問3(1)
解答例
h＝アップロードされたファイルの形式が拡張子と整合しているかをチェックする
ファイルのマジックバイト（magic bytes）によるコンテンツ検証。拡張子だけでなく、ファイル先頭バイトの実際のフォーマット（xlsx=PK..., pdf=%PDF- 等）を確認することで、JSファイルを.xlsx偽装してアップロードする攻撃を防ぐ。サーバ側でのContent-Type強制設定（application/octet-stream等）と組み合わせることも有効。


設問3(2)
解答例
i＝プロジェクト進捗管理
XSSの出力箇所を特定する問題。図2のHTMLはプロジェクト進捗管理画面のもの。タスク名を `<li>` 内に直接出力する際にHTMLエスケープをしていない箇所が脆弱点。SASTでは `period = (task.getName())` のような生データ出力パターンを検出できる可能性があるが、診断ツールXはこの出力コンテキストを理解できないため検知不可だった。


設問3(3)
解答例
出力時にエスケープ処理を施す
HTMLコンテキストへの出力エスケープ。`<` → `&lt;`、`>` → `&gt;`、`&` → `&amp;`、`"` → `&quot;` 等の変換により、挿入されたscriptタグをHTMLタグとして解釈させない。JavaのThymeleafやJSPではデフォルトでエスケープされる `th:text` / `<c:out>` を使用し、生HTML出力する場合は明示的に許可する設計にするのが鉄則。


設問4
解答例
j＝Sサービスの仕様を理解した専門家による脆弱性検査（ペネトレーションテスト）
DAST（診断ツールX）はリクエスト・レスポンスの自動分析で既知パターンを検出するが、アプリのビジネスロジック（「タスク名が進捗管理画面に出力される」という仕様）は理解できない。SAST（静的解析）はソースコードのパターン検出（esc漏れ等）に有効。専門家によるペネトレーションテスト（手動検査）は仕様理解を前提に機能横断的な攻撃シナリオを検証でき、今回のような論理的な脆弱性の発見に適している。SASTとペネトレーションテストは補完関係にある。
