問1　OAuthソーシャルログインの実装とセキュリティ

設問1(1)
解答例
利点：多要素認証の実装をSサービス側に用意しなくてよい 欠点：Tサービスの障害時にSサービスを利用できない
OAuthソーシャルログインのトレードオフ。利点は認証機能の外部委託によるMFA・パスワード管理の負担軽減。欠点は認証プロバイダ（IdP）への依存性・障害時の可用性低下・IdP側の情報漏えい影響。
設問1(3)
解答例
a＝ア　b＝イ　c＝ウ
OAuth 2.0 Authorization Code Flowの各ステップ。①認可リクエスト→②認可コード取得→③アクセストークン交換→④リソースアクセス の順序を理解する。stateパラメータによるCSRF対策も重要。
設問1(4)
解答例
（え）にstateパラメータの検証を追加する
OAuthのCSRF対策。stateパラメータをリクエスト時にセッションに保存し、コールバック時に照合することで、CSRF攻撃や認可コードインジェクション攻撃を防ぐ。PKCEも現在は推奨される。
設問2(2)
解答例
ファイルのアップロード（攻撃者） ファイルのダウンロード（攻撃者）
OAuthのアクセストークン悪用シナリオ。攻撃者がトークンを盗んだ場合、そのスコープ（権限）の範囲でAPIを呼び出せる。最小スコープの原則と短いトークン有効期限が被害を限定する。
設問3
解答例
S認証モジュールに利用者IDとパスワードを登録していないS会員
ソーシャルログインのみで登録したユーザ（パスワード未設定）。OAuthのみで認証していたユーザはパスワード認証経路が存在しないため、認証連携が壊れた場合にログイン手段を失う。パスワードリセット対応が必要。
設問4
解答例
TサービスでロールされたS会員のT-IDがSサービス内に登録されていることを確認する
OAuthのアカウント連携の整合性確認。Tサービス側でアカウントIDが変更・削除された場合、Sサービス側の紐付け情報が古くなる。定期的なIdPとの同期が必要。

問2　DNSセキュリティとDNSSEC

設問1(1)
解答例
A社公開WebサーバのIPアドレスの名前解決ができなくなる
DNS権威サーバへのDDoS攻撃の影響。権威DNSが応答不能になるとドメインの名前解決ができず、Web・メール等のサービスにアクセス不能になる。Anycast分散や複数の権威DNSで可用性を確保する。
設問1(2)
解答例
DNSリフレクション攻撃
DNSアンプリフィケーション攻撃。UDPの送信元を偽装（スプーフィング）してDNSサーバに大量のクエリを送ることで、応答を攻撃対象に集中させる。ANY/TXTレコードクエリは応答サイズが大きくなり増幅効果が高い。
設問1(5)
解答例
d＝ランダム化（ソースポートランダム化）
DNSキャッシュポイズニング対策。クエリのソースポートをランダム化することで、偽応答の正解率を下げる。Transaction IDのランダム化と組み合わせることで攻撃成功の確率を大幅に低下させる。
設問1(6)
解答例
e＝DNSSEC
DNSの応答を電子署名で保護する拡張仕様。RRSIG・DNSKEY・DS・NSECレコードを使って応答の完全性を検証できる。キャッシュポイズニングを根本的に防止できるが、鍵管理コストがかかる。
